Role switcher

A select dropdown and save button that changes the content of a dashboard page based on the user's chosen role view. This is a view preference, not an access control mechanism.

When to use this component

Use the role switcher on overview pages that serve multiple user types - for example, programme managers, finance officers and administrators - who each need different KPIs and actions from the same page.

When not to use this component

Do not use the role switcher for access control. If different users genuinely should not see certain data, implement proper role-based access control at the service level. The role switcher is a prototype pattern for testing different dashboard views, not a production security mechanism.

Relationship to the GOV.UK Design System

This is a new pattern not in the GOV.UK Design System. It uses the standard govuk-select and govuk-button components. The combined select-and-save control pattern does not currently appear in GDS.

How it looks in context

The role switcher sits in the top section of the dashboard overview page, above the stat cards and programme data. After applying a role view the page reloads and content is filtered to match the selected role.

Accessibility

  • WCAG 1.3.1 Info and Relationships: the <label> is explicitly associated with the <select> via matching for and id attributes.
  • WCAG 2.4.6 Headings and Labels: the label "Viewing as" clearly describes the purpose of the control.
  • Keyboard: the select is fully keyboard-operable using standard browser behaviour (Tab to focus, arrow keys to choose, Enter to confirm). The Apply button is reached by Tab and activated by Enter or Space.
  • On submit: the page reloads with focus managed to the main content heading so screen reader users are oriented after the role change.
  • WCAG 3.2.2 On Input: the role view does not change automatically on select - the user must press "Apply role view". This avoids unexpected context changes.

Research

During prototype testing, SP users with multiple responsibilities (e.g. both programme management and finance duties) wanted to view the same programme data through different lenses without logging out and back in. The role switcher was prototyped to test whether a view preference pattern could meet this need without requiring separate user accounts. [Research to be added]
