Accessible chart

Renders a Chart.js data visualisation paired with a visible data table, ensuring all chart data is accessible to screen reader users and users who cannot perceive colour.

When to use this component

Use the accessible chart on any page that presents trend data, comparisons or distributions visually. It is particularly suited to quarterly expenditure and units completion tracking in the SP service.

When not to use this component

For a single key metric, use a stat card instead. Do not use a chart to display data that is better expressed as a single sentence or as a summary list.

Relationship to the GOV.UK Design System

The GOV.UK Design System does not currently include a chart component. This pattern follows the W3C guidance on complex images and uses Chart.js for rendering. The canvas element is always paired with a data table so no information is conveyed by the chart alone.

Example

{# Accessible chart pattern: -
Show HTML 
<figure role="figure" aria-labelledby="chart-title-1">
  <figcaption id="chart-title-1" class="govuk-heading-s">
    Units completed by quarter
  </figcaption>

  <!-- canvas is aria-hidden: chart is purely visual -->
  <canvas id="my-chart" aria-hidden="true"
          width="600" height="300"></canvas>

  <!-- Data table is always in the DOM -->
  <table class="govuk-table">
    <caption class="govuk-visually-hidden">
      Units completed by quarter
    </caption>
    <thead class="govuk-table__head">...</thead>
    <tbody class="govuk-table__body">...</tbody>
  </table>
</figure>

<script>
new Chart(document.getElementById('my-chart'), {
  type: 'bar',
  data: {
    labels: ['Q1', 'Q2', 'Q3', 'Q4'],
    datasets: [{
      label: 'Units completed',
      data: [45, 62, 58, 82],
      backgroundColor: '#1d70b8'
    }]
  },
  options: {
    animation: { duration: window.matchMedia(
      '(prefers-reduced-motion: reduce)').matches ? 0 : 400
    }
  }
});
</script>

How it looks in context

Charts are typically placed on quarterly expenditure or programme overview pages to visualise trend data alongside the detailed data table.

Accessibility

  • WCAG 1.1.1 Non-text Content: the <canvas> element is aria-hidden="true". All data is present in the accompanying table which screen readers encounter directly.
  • WCAG 1.4.1 Use of Colour: chart bars use a single GDS blue (#1d70b8). Data differences are expressed in the table, not by colour alone. For multi-series charts, patterns or textures should supplement colour.
  • WCAG 1.4.11 Non-text Contrast: GDS blue (#1d70b8) on white achieves a 4.6:1 contrast ratio for graphical elements, exceeding the 3:1 minimum.
  • WCAG 2.3.3 Animation from Interactions: Chart.js animation duration is set to 0 when prefers-reduced-motion: reduce is active.
  • Screen reader behaviour: the <figure> element with role="figure" and aria-labelledby provides a labelled landmark. The data table is the primary data source.
  • Keyboard: no interactive controls on the chart itself. The "View data table" disclosure is keyboard operable via the native <details> element.

Research

SP users review expenditure and unit completion trends across multiple quarters. Visualising this data as a chart was identified as a way to surface trends quickly, but only if accompanied by a data table so the information remains available to all users. This pattern demonstrates that accessible charts do not require significant additional development effort. [Research to be added]
