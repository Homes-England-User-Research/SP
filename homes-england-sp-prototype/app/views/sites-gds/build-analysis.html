Sites (GDS)

Build analysis

This page documents how the GDS-only Sites page and Add new site flow were built. All elements use standard GOV.UK Design System components, patterns and layouts. The only agreed deviation is presenting multiple questions per page rather than one question per page.

{# -- Overview --------------------------------------------------------- #}

Overview

The GDS Sites flow is a pure GDS equivalent of the standard Sites flow, which uses custom stat cards and a two-column form layout. This version replaces every custom element with standard GDS components to demonstrate what is achievable with minimal departures from the Design System.

The flow consists of three pages: a landing page with the heading in its own two-thirds row, followed by quick links with descriptions in a two-thirds column and key statistics in a one-third sidebar, then a data table with pagination; and a two-step Add new site form with all fields in a single-column stack.

{# -- Pages covered ---------------------------------------------------- #}

Pages covered

  • Sites landing page (GDS) - heading row, two-thirds quick links with descriptions + one-third key statistics sidebar, data table, pagination
  • Add new site - Step 1 of 2 (GDS) - site details, location, contractor (single-column form)
  • Add new site - Step 2 of 2 (GDS) - site milestones (select dropdowns + date input, single-column)
{# -- GDS components used ---------------------------------------------- #}

GDS components used

Every component on these pages is an unmodified GOV.UK Design System component or standard GDS typography/layout class. No components have been extended, overridden or customised.

  • Service navigation - govukServiceNavigation(), unmodified. Programme-level navigation with Sites as current page.
  • Notification banner - govukNotificationBanner() with type: "success", unmodified. Shown after completing the Add new site flow.
  • Key statistics metrics - stacked govuk-body-s (label) above govuk-heading-m (value) separated by govuk-section-break dividers, placed in a govuk-grid-column-one-third sidebar alongside the two-thirds quick links column. Pure GDS typography and layout classes used as a lightweight replacement for stat cards. Same pattern used on the GDS Overview page key information sidebar.
  • Table - govuk-table classes, unmodified. scope="col" on all column headers. Numeric column uses govuk-table__header--numeric and govuk-table__cell--numeric.
  • Tag - govukTag() with colour variants (govuk-tag--blue, govuk-tag--yellow, govuk-tag--grey, govuk-tag--green), unmodified. Site status indicators.
  • Pagination - govukPagination(), unmodified. Previous/Next links with numbered page items.
  • Breadcrumbs - govukBreadcrumbs(), unmodified. Used on all pages with appropriate hierarchy.
  • Text input - govukInput(), unmodified. Single-column stacked layout with GDS width classes (govuk-input--width-10, govuk-!-width-two-thirds).
  • Select - govukSelect(), unmodified. All dropdowns in both form steps.
  • Date input - govukDateInput(), unmodified. Forecast practical completion date in step 2.
  • Button - govukButton(), unmodified. Primary and secondary variants in govuk-button-group.
  • Grid layout - govuk-grid-row with govuk-grid-column-two-thirds + govuk-grid-column-one-third sidebar for landing page metrics; govuk-grid-column-two-thirds for form pages. Standard GDS grid layout.
  • Typography - govuk-heading-xl, govuk-caption-xl, govuk-heading-m, govuk-body-l. All standard GDS classes.
  • Spacing - all spacing uses govuk-!-margin-* utility classes. No custom pixel values.
{# -- Custom departures ------------------------------------------------ #}

Custom departures

Multi-question pages (agreed deviation)

Departure: The Add new site form presents multiple related fields on a single page across two steps, rather than one question per page. Unlike the standard Sites version, all fields are in a single-column stack rather than a two-column grid.

Justification: SP users are experienced data entry professionals. The fields are grouped by subject area (site details, location, contractor in step 1; milestones in step 2). A single-question-per-page approach would create an excessively long journey. GDS guidance explicitly allows grouping questions: "there are times when it makes sense to group a few questions on the same page" (GDS question pages pattern).

No other custom departures. Every other element uses standard GDS components. No custom CSS classes, no custom Sass, no inline styles and no custom JavaScript.

{# -- Design trade-offs ------------------------------------------------ #}

Design trade-offs

Stat cards replaced with stacked typography in a sidebar

The standard Sites page uses three custom stat cards in a CSS grid to display Total sites, Total homes and Completed sites. The GDS version replaces these with stacked govuk-body-s / govuk-heading-m pairs separated by govuk-section-break dividers, placed in a govuk-grid-column-one-third sidebar. Quick links sit in the adjacent govuk-grid-column-two-thirds primary column. This is the same two-thirds + one-third pattern used on the GDS Overview page. The data is identical but the visual prominence is reduced - metrics are read vertically in a sidebar rather than in a scannable grid with bordered cards. For users who scan dashboards quickly, this is a regression. For GDS compliance, it is the correct approach.

Filters removed

The standard Sites page uses govukDetails wrapping govukCheckboxes with custom auto-redirect JavaScript to filter sites by status and region. While the individual components (Details, Checkboxes) are standard GDS, the composition into a filter pattern with auto-submitting JS is bespoke - the GOV.UK Design System has no documented filter pattern. To maintain a pure GDS build, the filters have been removed entirely. Users see all sites with pagination only. This is a functional regression but eliminates the only custom JavaScript on the page.

{# -- Accessibility ---------------------------------------------------- #}

Accessibility

Because every component on these pages is a standard, unmodified GOV.UK Design System component, the accessibility characteristics are inherited directly from the Design System. GOV.UK Frontend components are tested against WCAG 2.2 AA by the GDS team.

Reduced testing burden compared to the standard Sites version. The standard version requires a full 12-row assistive technology matrix to validate the custom stat cards. The GDS version needs only standard GDS component testing:

  • Run axe DevTools to confirm zero violations - expected to pass as all components are standard GDS
  • Run Lighthouse accessibility audit - expected to score 100
  • Verify keyboard navigation - all GDS components are natively keyboard accessible
  • Verify screen reader announcements - GDS tables, tags, summary lists and form components have built-in ARIA support

The multi-question layout is the one area worth manual confirmation. While all form fields use standard govukInput() and govukSelect() macros with associated labels, the grouping of multiple fields with section headings should be checked with a screen reader to confirm logical reading order.
