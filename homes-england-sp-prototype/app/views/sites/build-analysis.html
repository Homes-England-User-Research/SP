Sites

Build analysis

This page documents how the Sites landing page and Add new site flow were built, how they align with the GOV.UK Design System and WCAG 2.1 AA, all custom departures with their justifications, and predicted accessibility tool outcomes.

{# -- Overview --------------------------------------------------------- #}

Overview

The Sites flow consists of three pages: a landing page with stat cards, status and region filters and a data table; and a two-step Add new site form that collects site details, location, contractor information and milestones. The flow follows the IMS (Investment Management System) pattern of dense, data-rich pages rather than the standard GDS one-question-per-page pattern, which is not appropriate for this use case.

All GDS components used on these pages are used without modification unless listed under Custom departures below.

{# -- Pages covered ---------------------------------------------------- #}

Pages covered

  • Sites landing page - stat cards, Details + Checkboxes filters, data table, pagination
  • Add new site - Step 1 of 2 - site details, location, contractor (multi-field form)
  • Add new site - Step 2 of 2 - site milestones (select dropdowns + date input)
{# -- GDS alignment ---------------------------------------------------- #}

GDS alignment

  • Notification banner - govukNotificationBanner() with type: "success", unmodified. Shown on the Sites landing page after completing the Add new site flow.
  • Breadcrumbs - govukBreadcrumbs(), unmodified. Used on all three pages with appropriate hierarchy.
  • Tag - govukTag() with colour variants (govuk-tag--blue, govuk-tag--yellow, govuk-tag--grey, govuk-tag--green), unmodified. Used in the Sites table Status column to indicate site lifecycle state.
  • Table - govuk-table classes throughout, with govuk-table__header--numeric and govuk-table__cell--numeric for the Homes column. scope="col" on all column headers. Table caption removed — the table's purpose is clear from the page heading and surrounding context; the H1 "Sites" and the filter controls directly above the table provide sufficient context.
  • Pagination - govukPagination(), unmodified. Previous/Next links with numbered page items.
  • Text input - govukInput(), unmodified. Used in the Add new site form for Site ID, Site name, Local authority, Operating area, Postcode, coordinates and Contractor fields.
  • Select - govukSelect(), unmodified. Used in both form steps for Type of site, Rural area, Region, Processing route, Regeneration site, Type of contractor (step 1) and Ownership status, Planning status, Building contract status, Start on site status (step 2).
  • Date input - govukDateInput(), unmodified. Used in step 2 for Forecast practical completion date. Day/Month/Year pattern with govuk-input--width-2 and govuk-input--width-4 classes.
  • Button - govukButton(), unmodified. Primary (Continue, Save site) and secondary (Save as draft) variants. Save as draft uses href to link back to Sites without submitting.
  • Grid - govuk-grid-row with govuk-grid-column-one-half for paired form fields in Add new site step 1. Standard GDS 12-column grid, unmodified.
  • Spacing - all spacing uses govuk-spacing() tokens or govuk-! utility classes. No custom pixel values.
  • Typography - govuk-heading-xl with govuk-caption-xl for step indicators ("Step 1 of 2", "Step 2 of 2"). govuk-heading-m for form section headings. All standard GDS classes.
  • Numeric columns - govuk-table__header--numeric and govuk-table__cell--numeric on the Homes column, per GDS numeric table guidance.
{# -- Custom departures ------------------------------------------------ #}

Custom departures

{# -- Stat cards ------------------------------------------------------- #}

Stat cards (3-column grid)

Departure: Three metric cards (Total sites, Total homes, Completed sites) displayed in a 1x3 CSS grid using .stat-card-grid--3col. No GDS equivalent exists.

Justification: Reuses the stat card component already established on the Overview page. The 3-column variant is a narrower layout suited to pages where fewer metrics are displayed. Built entirely from GDS design tokens. The coloured top border convention is used across HMRC and ONS internal tools. A visually hidden heading ("Key site statistics") wraps the cards in a labelled <section> for screen reader landmark navigation (WCAG 1.3.1).

Accessibility: Reflows to single column at 400% zoom (WCAG 1.4.10 Reflow). No horizontal scroll at any zoom level.

{# -- Status and region filters ----------------------------------------- #}

Filters (Details + Checkboxes)

Not a departure. Two filter controls — "Filter by status" and "Filter by region" — sit directly above the table in a quarter-width grid layout (govuk-grid-column-one-quarter). Each uses standard GDS govukDetails wrapping govukCheckboxes with the govuk-checkboxes--small variant. Both components are unmodified. The filters are positioned outside the <table> element, as page-level controls rather than embedded in the table header.

Why this composition: The standard checkboxes component stacks vertically, consuming significant page height. SP users need to see the data table as quickly as possible without scrolling past a tall filter panel. Wrapping the checkboxes inside Details components keeps the filters collapsed by default, reducing the vertical footprint to a single line each. Users expand them when they need to change the filter. This is pure GDS with no custom code.

Accessibility: Both govukDetails and govukCheckboxes are fully accessible out of the box. The Details component uses the native HTML <details>/<summary> elements, which are keyboard-operable and announced correctly by all screen readers. The checkboxes inside inherit full GDS accessibility including associated labels and focus styles.

{# -- Multi-field form ------------------------------------------------- #}

Multi-field form (Add new site)

Departure: The Add new site form presents multiple related fields on a single page across two steps, rather than one question per page. Fields are arranged in a two-column grid layout using govuk-grid-row and govuk-grid-column-one-half.

Justification: SP users are experienced data entry professionals already familiar with the IMS system. The fields are grouped by subject area (site details, location, contractor in step 1; milestones in step 2). A single-question-per-page approach would create an excessively long journey for a user completing this task routinely. GDS guidance explicitly allows grouping questions where appropriate: "there are times when it makes sense to group a few questions on the same page" (GDS question pages pattern).

Accessibility:

  • DOM order follows a logical top-to-bottom, left-to-right reading sequence (WCAG 1.3.2) - the two-column layout is achieved with CSS Grid only
  • All form fields use standard govukInput() and govukSelect() macros with associated <label> elements (WCAG 1.3.1)
  • Section headings (govuk-heading-m) group related fields visually and in the DOM - "Site details", "Site location", "Contractor details", "Site milestones"
  • Step indicator uses govuk-caption-xl ("Step 1 of 2") providing progress context (WCAG 2.4.8)
  • At 400% zoom, columns collapse to single column - no horizontal scroll (WCAG 1.4.10)
{# -- Implementation details ------------------------------------------- #}

Implementation details

Session-based multi-step form

The Add new site form uses the GOV.UK Prototype Kit's session storage. Step 1 saves all fields to req.session.data['newSite'] and redirects to step 2. Step 2 merges its fields into the same session object and sets a newSiteAdded flag before redirecting to /sites?success=true. The Sites landing page detects the flag and prepends the new site to the table. This follows the standard Prototype Kit POST-redirect-GET pattern.
