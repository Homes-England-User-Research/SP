Allocate home types

Build analysis

This page documents how the Allocate home types flow was built, how it aligns with the GOV.UK Design System and WCAG 2.1 AA, and predicted accessibility tool outcomes. This flow uses zero custom components - every element is a standard GDS component or pattern.

{# -- Overview --------------------------------------------------------- #}

Overview

The Allocate home types flow allows SP users to manage which home types from their organisation are included at a specific site. It follows the GDS "Add to a list" pattern: a main page showing currently allocated items with remove actions, a separate page for adding new items via checkboxes, and a confirmation page before removal.

Every GDS component used on these pages is used without modification. There are no custom departures.

{# -- Pages covered ---------------------------------------------------- #}

Pages covered

  • Allocate home types - main page with allocated home types table, "Add home types" and "Create new home type" buttons
  • Add home types - checkbox selection page listing unallocated home types
  • Remove home type - confirmation page with summary list, warning text and Yes/No radios
{# -- GDS alignment ---------------------------------------------------- #}

GDS alignment

All components are standard GDS, used via their official Nunjucks macros with no class overrides or modifications.

  • Breadcrumbs - govukBreadcrumbs(), unmodified. Used on all three pages with appropriate hierarchy (Sites > Site summary > Allocate home types).
  • Notification banner - govukNotificationBanner() with type: "success", unmodified. Shown after adding or removing home types with a count of items added.
  • Table - govukTable(), unmodified. Used on the main page to display allocated home types with columns for ID, description, building type, bedrooms, persons and a Remove action link. Visually hidden caption for screen readers.
  • Button - govukButton(), unmodified. Primary variant for "Add home types" and "Add selected home types"; secondary variant for "Create new home type" and "View build analysis". Uses govuk-button-group for paired button layout.
  • Checkboxes - govukCheckboxes(), unmodified. Used on the Add page with fieldset legend as page heading (isPageHeading: true). Each checkbox has a hint showing building type, bedrooms and persons.
  • Radios - govukRadios(), unmodified. Used on the Remove confirmation page for the Yes/No question.
  • Summary list - govukSummaryList(), unmodified. Used on the Remove confirmation page to show full details of the home type being removed.
  • Warning text - govukWarningText(), unmodified. Used on the Remove confirmation page to reassure users that removal is reversible.
  • Back link - govukBackLink(), unmodified. Used on the Add and Remove pages to return to the main allocate page.
  • Inset text - standard govuk-inset-text class, unmodified. Used for the empty state when no home types are allocated.
  • Grid - govuk-grid-row with govuk-grid-column-two-thirds for text content. Table sits full-width to accommodate its columns. Standard GDS 12-column grid.
  • Typography - govuk-heading-xl with govuk-caption-xl for "Site 1234". govuk-heading-m for section heading. All standard GDS classes.
  • Spacing - all spacing uses govuk-!-margin-* and govuk-!-padding-* utility classes. No custom pixel values.
{# -- GDS patterns ----------------------------------------------------- #}

GDS patterns used

Add to a list

The flow follows the GDS Add multiple things pattern. The main page shows a list of items (allocated home types) with the ability to remove individual items and a button to add more. The add page uses checkboxes to allow batch selection. After submission, the user returns to the main page with a success notification.

Question page

The Add home types page follows the Question pages pattern exactly: a single question as the page heading (using isPageHeading: true on the fieldset legend), hint text providing context, and a submit button. The two-thirds column width constrains reading line length per GDS guidance.

Confirmation before destructive action

The Remove page follows the Confirmation pages pattern: showing full details of what will be affected (via summary list), a warning explaining consequences, and a Yes/No radio for explicit confirmation. This prevents accidental removal.

{# -- Custom departures ------------------------------------------------ #}

Custom departures

None. This flow uses exclusively standard GDS components and patterns. No custom CSS classes, no custom JavaScript, no custom components. All styling comes from GDS utility classes and component defaults.

{# -- Implementation details ------------------------------------------- #}

Implementation details

Session-based allocation state

The allocated home types are stored in req.session.data['allocatedHomeTypes'] as an array of home type IDs. On first visit, the session is seeded with 4 default allocations. Adding home types appends IDs to this array; removing splices them out. The full home type data (description, building type, bedrooms, persons) lives in a server-side array and is filtered by the session IDs on each request. This follows the standard Prototype Kit session pattern.

POST-redirect-GET

Both the Add and Remove forms follow the POST-redirect-GET pattern. After processing, the server redirects to the main page with query parameters (?success=3 or ?removed=true) that trigger the appropriate notification banner. This prevents duplicate submissions on browser refresh.

Accessible Remove links

Each Remove link in the table includes a govuk-visually-hidden span containing the home type description. This means screen readers announce "Remove 2B4P House - Social Rent" rather than just "Remove", distinguishing between multiple Remove links on the page (WCAG 2.4.4 Link Purpose in Context).
