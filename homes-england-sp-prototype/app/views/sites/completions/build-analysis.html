Completions

Build analysis

This page documents how the Completions journey was built, how it aligns with the GOV.UK Design System and WCAG 2.1 AA, all custom departures with their justifications, and predicted accessibility tool outcomes.

{# -- Overview --------------------------------------------------------- #}

Overview

The Completions journey is a 7-screen flow where SP users record phased home completions after practical completion. Users create completion phases, assign tenure home types from the programme's allocated home type list, enter rent and sales data per tenure type, record individual home addresses with RCGF values, enter phase-level costs and contributions, and submit for Homes England review.

The journey follows the IMS (Investment Management System) pattern of dense, data-rich pages with multiple related fields per screen. This is intentional for SP professional users who are experienced data entry specialists. All GDS components used are unmodified unless listed under Custom departures below.

{# -- Pages covered ---------------------------------------------------- #}

Pages covered

  • Completions landing -phase table, action links, success banner, pagination
  • Add a completion phase -phase details input, practical completion date
  • Phase hub -5-tab dashboard (phase details form, tenure home type list table, phase costs form, phase contributions form, review and submit), gated submission
  • Add tenure home type -allocated home type select with progressive reveal, tenure type select, conditional rent terms radios
  • Rent and sales -tenure-type-conditional fields (Shared Ownership: 5 fields, Affordable/Social Rent: 2 fields, Rent to Buy: 3 fields)
  • Add home addresses -inline editable table with dynamic row management, RCGF calculation
  • Submit phase -read-only review, further phases radios, certification checkbox, comments
{# -- GDS alignment ---------------------------------------------------- #}

GDS alignment

  • Breadcrumbs -govukBreadcrumbs(), unmodified. Used on the completions landing page with Sites > [Site name] hierarchy.
  • Back link -govukBackLink(), unmodified. Used on all sub-pages (add phase, phase hub, tenure home type screens, submit) to provide escape back to the parent screen.
  • Notification banner -govukNotificationBanner() with type: "success", unmodified. Shown on the landing page after successful phase submission. Also used on the rent and sales page for the conversion note.
  • Table -govukTable() and manual govuk-table markup throughout. scope="col" on all column headers. govuk-table__header--numeric and govuk-table__cell--numeric for numeric columns (homes count, grant amounts).
  • Tag -govukTag() with colour variants (govuk-tag--blue for in progress, govuk-tag--green for submitted/complete, govuk-tag--yellow for incomplete, govuk-tag--red for readiness check failures). Text always accompanies colour (WCAG 1.4.1).
  • Tabs -govukTabs(), unmodified. Phase hub uses 5 tabs: Phase details, Tenure home type list, Phase costs, Phase contributions, Review and submit. Auto-initialised by govuk-frontend with full keyboard support (arrow key navigation, Enter/Space activation). Tab panel IDs must not conflict with any input IDs inside panels to avoid scroll-jump issues (see Implementation details).
  • Summary list -govukSummaryList(), unmodified. Used for: phase header (status, homes, ID), read-only submitted state on all tabs, derived totals (total costs, RCGF, total contributions) displayed as key-value pairs within forms, readiness check in the Review and submit tab (4 rows with green/red status tags), and phase details review on the submit page.
  • Text input -govukInput(), unmodified. Used throughout with prefix: { text: "£" } for currency fields and suffix: { text: "%" } for percentage fields. All numeric inputs use type="text" inputmode="numeric" (never type="number" to avoid JAWS spin button announcement).
  • Select -govukSelect(), unmodified. Used for owner, contractor, and tenure type selects. Default placeholder options use disabled: true with selected conditional on empty value.
  • Date input -govukDateInput(), unmodified. Day/Month/Year pattern with govuk-input--width-2 and govuk-input--width-4 classes. Used for practical completion date.
  • Radios -govukRadios(), unmodified. Used on the submit page for further phases question and on the add tenure home type page for rent terms.
  • Checkboxes -govukCheckboxes(), unmodified. Used on the submit page for the certification declaration.
  • Textarea -govukTextarea(), unmodified. Used on the submit page for optional comments.
  • Error summary -govukErrorSummary(), unmodified. Displayed at the top of form pages when validation fails. Error messages link to the relevant field.
  • Warning text -govukWarningText(), unmodified. Used for contributions/costs mismatch warning and for the submitted-state lock message.
  • Button -govukButton(), unmodified. Primary (Save, Submit), secondary (Add address, Copy last address, Unsubmit phase, Add tenure home type) and disabled variants used throughout.
  • Pagination -govukPagination(), unmodified. Static pagination on the completions landing page.
  • Spacing -all spacing uses govuk-! utility classes. No custom pixel values.
  • Typography -govuk-heading-xl with govuk-caption-xl for site context. Section headings use govuk-heading-m and govuk-heading-s. All standard GDS classes.
{# -- Custom departures ------------------------------------------------ #}

Custom departures

{# -- Inline editable address table ------------------------------------ #}

Inline editable address table

Departure: The Add home addresses page uses an inline editable table where each row contains form inputs inside table cells. Users can add rows, copy the last row, and delete rows dynamically without page reload. A running RCGF total updates live as values change.

Justification: Reuses the inline editable table pattern already established on the Site home forecasts page. SP users enter multiple addresses per tenure home type -a one-address-per-page approach would create an impractical number of page transitions for sites with 50+ homes. The IMS system uses a similar tabular entry pattern. GDS guidance permits complex data entry tools where they serve the user need: "there are times when it makes sense to group a few questions on the same page" (GDS question pages pattern).

Accessibility:

  • Every input has aria-label="Home [n] - [Column name]" matching the visible column header text for Dragon NaturallySpeaking compatibility
  • aria-labels are dynamically updated when rows are added or deleted via the reindexRows() function
  • RCGF total uses aria-live="polite" so screen readers announce the updated total (WCAG 4.1.3)
  • Currency prefix uses aria-hidden="true" to avoid duplicate "pound" announcements
  • All numeric inputs use type="text" inputmode="numeric" -never type="number" (JAWS spin button issue)
  • New row creation moves focus to the first input of the new row for keyboard users
  • Delete requires at least one row to remain -prevents an empty, unrecoverable state
  • Table wrapped in .sp-table-wrapper for horizontal scroll on narrow viewports
{# -- Implementation details ------------------------------------------- #}

Implementation details

Session-based nested data structure

The completions journey stores deeply nested session data: completions.phases[].tenureHomeTypes[].addresses[]. Each POST handler manually updates the relevant nested object rather than relying on the Prototype Kit's flat auto-save behaviour. Helper functions (ensureCompletionsSession(), findPhase(), findTenureHomeType()) encapsulate session traversal and initialisation. Phase IDs are generated as CP-001, CP-002 etc.; tenure home type IDs as THT-001, THT-002.

Live auto-calculation (Phase costs)

The Phase costs tab includes inline JavaScript that auto-calculates the total costs value (acquisition + work + on costs) as the user types. The total is displayed in a govukSummaryList row with the value wrapped in a <span id="total-costs-value" aria-live="polite">. This follows the same pattern established in the Site home forecasts editable table. The RCGF total on the addresses page uses the same approach. Both are IIFE-wrapped vanilla JavaScript with event delegation.

Review and submit tab (readiness check and gated submission)

The phase hub's 5th tab ("Review and submit") consolidates the readiness check and submission controls. Four readiness checks are calculated server-side: phase details complete (owner and contractor set), tenure home types complete (at least one valid THT with rent/sales data and addresses), phase costs complete (all 4 cost fields populated), and phase contributions complete (total contributions matches total costs). Each check is displayed as a govukSummaryList row with a green "Complete" or red "Incomplete" / "Totals don't match" govukTag. The submit button is rendered as a disabled <button> with aria-disabled="true" until all checks pass, at which point it becomes an active link to the submit review page. When the phase is submitted, the tab shows a warning text and an "Unsubmit phase" secondary button.

Tab reselection after POST

Each tab form POSTs to its own route and redirects back to the phase hub. The GDS tabs component responds to URL hash fragments (#phase-details, #phase-costs, etc.), so the redirect includes the appropriate fragment to reselect the correct tab after save. Delete tenure home type redirects with #tenure-home-type-list to return the user to tab 2. The submit page back link targets #review-and-submit to return the user to the 5th tab.

Route ordering

All completions routes are defined before the wildcard /sites/:siteId route to prevent Express matching "completions" as a :siteId parameter. Within the completions block, static routes (/completions/add) are defined before parameterised routes (/completions/:phaseId). The phase hub GET route is last because its :phaseId wildcard would match other paths.
