Site summaries

Build analysis

This page documents how the three site summary page options were built, how they align with the GOV.UK Design System and WCAG 2.1 AA, all custom departures with their justifications, and predicted accessibility tool outcomes.

{# -- Overview --------------------------------------------------------- #}

Overview

The site summary is the CRM-style detail view that users land on when clicking into a specific site from the Sites table. Three layout options have been built to test different approaches in user research:

  • Option 1 — sidebar layout with "Site related actions" links + metric cards + tabs
  • Option 2 — full-width layout with grouped task list progress + metric cards + tabs
  • Option 3 — pure GDS version with key information sidebar (no custom components)

All three options share the same tab content: Site progress (task list), Site details (form fields matching the Add new site step 1) and Site milestones (form fields matching step 2). The difference is in how the page-level layout, metrics and navigation are presented.

{# -- GDS alignment ---------------------------------------------------- #}

GDS alignment

  • Breadcrumbs — govukBreadcrumbs(), unmodified. Links back to Sites landing page.
  • Tabs — govukTabs(), unmodified. Three panels: Site progress, Site details, Site milestones. Auto-initialised by govuk-frontend JavaScript. Falls back to stacked sections without JS.
  • Task list — govukTaskList(), unmodified. Used in the Site progress tab to show lifecycle stages with status indicators. Option 1 uses a flat list; Option 2 uses grouped sections with numbered headings.
  • Text input — govukInput(), unmodified. Same fields as Add new site step 1 (Site ID, Site name, Local authority, etc.).
  • Select — govukSelect(), unmodified. Same dropdowns as Add new site steps 1 and 2.
  • Date input — govukDateInput(), unmodified. Forecast practical completion date on Site milestones tab.
  • Button — govukButton(), unmodified. Save changes (primary) and Cancel changes (secondary).
  • Grid — govuk-grid-row with govuk-grid-column-one-third / govuk-grid-column-two-thirds for sidebar layouts (Options 1 and 3). Standard GDS 12-column grid.
  • Typography — govuk-heading-xl with govuk-caption-xl for site ID. Section headings use govuk-heading-m.
{# -- Custom departures ------------------------------------------------ #}

Custom departures

{# -- Stat cards ------------------------------------------------------- #}

Stat cards — Options 1 and 2 only

Departure: Three metric cards (Total grant attributable, Site status, Grant funded homes) displayed using the custom .stat-card component in a 3-column grid (.stat-card-grid--3col). No GDS equivalent exists.

Justification: Reuses the stat card component established on the Overview and Sites landing pages. At-a-glance metrics are critical for CRM-style detail views where users need to assess site health quickly. Built entirely from GDS design tokens. A visually hidden heading ("Key site statistics") wraps the cards in a labelled <section> for screen reader landmark navigation (WCAG 1.3.1).

GDS alternative (Option 3): Option 3 replaces these cards with stacked GDS typography in a one-third sidebar, using govuk-body-s labels and govuk-heading-m values with govuk-section-break dividers. This is the same pattern used on the GDS Sites landing page (/sites-gds).

{# -- Sidebar links ---------------------------------------------------- #}

Sidebar navigation — Option 1 only

Not a departure. Option 1 includes a sidebar with "Site related actions" links using a standard <ul class="govuk-list"> with govuk-link anchors. This is standard GDS markup.

Design rationale: Provides quick access to common site management tasks without leaving the summary page context. This mirrors the IMS pattern where related actions are always visible alongside the detail view.

{# -- Grouped task list ------------------------------------------------ #}

Grouped task list — Option 2 only

Not a departure. Option 2 uses the standard govukTaskList() macro multiple times with numbered section headings (govuk-heading-m) and descriptions (govuk-body) above each group. The task list component itself is unmodified — the grouping is achieved through standard HTML headings and paragraphs placed between separate task list invocations.

{# -- Multi-field form ------------------------------------------------- #}

Multi-field form tabs — all options

Departure: The Site details and Site milestones tabs present multiple related fields on a single tab panel, rather than one question per page. This is the same departure documented in the Sites flow build analysis.

Justification: SP users are experienced data entry professionals. The fields are the same as the Add new site form steps, displayed in a single-column layout within tab panels. Section headings group related fields. GDS guidance allows grouping questions where appropriate.

{# -- Implementation details ------------------------------------------- #}

Implementation details

Route and data enrichment

All three options use the same data pattern: GET /sites/:siteId (and /summary-2, /summary-3 variants) look up the site from the seedSites array by ID, then enrich the site object with mock form field values (type of site, contractor details, milestone statuses, forecast completion date). This mirrors how a real service would fetch site data from an API and pass it to the template.

Tab content reuse

The Site details and Site milestones tab content is defined once per template using Nunjucks {% set %} blocks, then passed to govukTabs(). The form fields match those on the Add new site pages but are pre-populated with the enriched site data. Field widths match the GDS version at /sites-gds/add/step-1.
