Site home forecasts

Build analysis

This page documents how the Site home forecasts page was built, how it aligns with the GOV.UK Design System and WCAG 2.1 AA, all custom departures with their justifications, and predicted accessibility tool outcomes.

{# ── Overview ───────────────────────────────────────────────────────────── #}

Overview

The Site home forecasts page allows Strategic Partnerships (SP) users to enter and review home completion forecasts per tenure type, across two funding tiers. It follows the IMS (Investment Management System) pattern of dense, data-rich inline editing rather than the standard GDS one-question-per-page pattern, which is not appropriate for this use case.

All GDS components used on the page are used without modification unless listed under Custom departures below.

{# ── GDS alignment ──────────────────────────────────────────────────────── #}

GDS alignment

  • Breadcrumbs — govukBreadcrumbs macro, unmodified
  • Page heading — govuk-heading-xl with govuk-caption-l, standard GDS pattern
  • Table markup — govuk-table, govuk-table__header, govuk-table__cell, standard GDS classes throughout
  • Numeric columns — govuk-table__header--numeric and govuk-table__cell--numeric per GDS numeric table guidance
  • Table captions — govuk-table__caption govuk-table__caption--m for programmatic table association (WCAG 1.3.1); preferred over an external <h2> which has no native association with the table element
  • Form inputs — govuk-input with type="text" inputmode="numeric" per GDS recommendation; type="number" causes JAWS to announce "spin button"
  • Save button — govuk-button, unmodified
  • Secondary button — govuk-button govuk-button--secondary, unmodified
  • Button group — govuk-button-group, standard GDS pattern
  • Spacing — all spacing uses govuk-spacing() tokens or govuk-! utility classes; no raw pixel values in custom Sass
  • Deficit indicator — govuk-tag govuk-tag--red inside <strong>, conditionally rendered in a table cell when Homes remaining falls below zero. The GDS tag component is documented for indicating status, and govuk-tag--red is the approved variant for problematic or urgent states. Using a tag in a table cell is an explicitly supported pattern in the GDS design system.
  • Page width — max-width: 1020px; GDS states the default is 1020px and explicitly permits widening for content that requires it
{# ── Custom departures ──────────────────────────────────────────────────── #}

Custom departure

Editable inline table pattern

Departure: The pattern of placing editable inputs directly inside a data table, with an auto-calculated read-only column, does not exist in the GOV.UK Design System.

Justification: The page presents data for 12 tenure types across 2 funding tiers, where each row has a related forecast input, a read-only actual value, and an auto-calculated remaining value. This is a tabular relationship between values — not a sequence of unrelated questions. Placing these as separate pages would destroy the relational context that makes the data meaningful to users. GDS already supports multiple related inputs on a single page through the fieldset component (for example, address fields and date inputs). The GDS question pages pattern explicitly acknowledges exceptions for complex data entry tools. The pattern is implemented using entirely standard GDS table markup and govuk-input components — no GDS styles are overridden. Users are professional case managers at housing associations and developers bidding under the Affordable Homes Programme, accessing the service via an account system.

{# ── Implementation details ─────────────────────────────────────────────── #}

Implementation details

The following are implementation choices that flow from the editable table pattern. None override GDS components — they are additive adjustments to make the pattern work correctly and remain WCAG 2.1 AA compliant.

Auto-calculated "Homes remaining" column

Client-side JavaScript recalculates remaining homes (forecast minus actuals) in real time as the user types. The column is also server-calculated on page load so it works without JavaScript. aria-live="polite" (WCAG SC 4.1.3) is applied to each calculated cell so screen readers announce the updated value without interrupting ongoing speech.

Column header base alignment

vertical-align: bottom is applied to column header cells so all headers sit flush to the table body regardless of whether they wrap to one or two lines. GDS does not set vertical-align on header cells, so this is additive with no conflict.

margin-bottom: 0 on inputs inside table cells

GDS inputs carry a default margin-bottom designed for stacked form layouts. Inside a table cell this adds unintended height to every row. Removing it achieves the expected density without affecting focus styles or touch target size, which are determined by padding not margin.
